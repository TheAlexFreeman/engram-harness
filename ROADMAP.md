# Engram Harness — Project Roadmap

A merged repository combining the **Engram** memory system with the **Harness**
agent loop. The harness serves as Engram's session-management frontend; Engram
provides the harness with durable, cross-session memory and activity history.

> **Status (2026-04-27).** The original Phases 0–3 of this roadmap are fully
> shipped, and Phases 4–7 are largely superseded by the contemporary
> work-stream documented in [docs/improvement-plans-2026.md](docs/improvement-plans-2026.md).
> That document is the **active** sequencing for ongoing harness development
> — this file remains as the motivation/principles reference and a record of
> the original phase plan with shipped state annotated. See the [Phases](#phases)
> section below for the per-phase status, and the improvement-plans doc for

> all shipped; A2, B2, B3, D1, D2, E1 open).

## Motivation

Engram is a model-portable, human-legible, version-controlled, adaptively
self-organizing memory layer for AI agents. It stores knowledge as plain markdown
files in a git repository — inspectable, diffable, owned by the user. Its design
is built around a feedback loop: ACCESS.jsonl tracks what gets retrieved and how
helpful it was, aggregation amplifies high-value content and retires low-value
content, and the system improves through use rather than just accumulation.

That feedback loop has been starved of input. Activity tracking requires
comprehensive session data — which tool calls happened, which files were read,
what was helpful, what cost what — and no existing integration has provided that
at sufficient volume or granularity. The self-organizing features (aggregation,
emergent categorization, compression) are architecturally complete but data-poor.

The harness produces exactly the signal Engram needs: JSONL traces of every tool
call, model turn, token cost, timing, and outcome for every session it runs.
Merging the two turns the harness into a comprehensive data collection frontend
for Engram's memory lifecycle, and gives the harness access to Engram's semantic
retrieval, persistent plans, and cross-session continuity.

The integrated project makes Engram's most interesting properties operational:
skill emergence from observed task clusters, retrieval optimization from
helpfulness evidence, and progressive compression grounded in actual usage data
rather than time-decay heuristics alone.

## Design Principles

These principles govern the integrated project. They derive from Engram's core
architectural commitments (documented in `HUMANS/docs/CORE.md` and
`HUMANS/docs/DESIGN.md`) and shape how every phase is implemented.

### 1. Files over APIs

The most important architectural decision in Engram: memory is markdown files in
a git repository. Not a database. Not an embedding store. Not a platform feature.

This applies unconditionally to harness integration. Trace bridge output is plain
markdown with YAML frontmatter, committed to git. Session records are `.md` files
in `activity/`. Skill proposals are `.md` files in `skills/_proposed/`. Everything
the harness writes to Engram is human-readable, human-editable, diffable, and
version-controlled. The harness never writes to Engram through a path that
bypasses git.

### 2. Context efficiency as first-class concern

Every piece of memory competes for space in the agent's context window. Every
token spent on system overhead is a token not spent on the user's task.

The harness must respect Engram's context loading manifests and token budgets.
`EngramMemory.start_session()` follows the compact returning path (~3,000–7,000
tokens), not a full bootstrap. It loads HOME.md, user portrait, activity summary,
and relevant scratchpads — then drills down only when the task demands it.
Session records generated by the trace bridge must be compact enough to be useful
as retrieval targets without dominating the context window.

### 3. Progressive compression

Engram uses a compression hierarchy that mirrors human memory: recent events are
detailed, older events compress into themes, and only significant moments from
the distant past remain individually accessible.

Harness trace data participates in this hierarchy. Individual session records are
detailed (tool calls, costs, outcomes). Monthly summaries abstract to themes and
patterns. Yearly summaries abstract to trajectories. The trace bridge generates
leaf-level records; Engram's existing aggregation pipeline handles compression
upward. The harness does not maintain its own parallel history — it feeds the
existing hierarchy.

### 4. Forgetting is maintenance

Memory without forgetting degrades. The trace bridge will generate substantial
data — potentially a session record per run, dozens of ACCESS entries per session,
trace spans for every tool call. Without explicit curation, this pollutes the
system and makes retrieval worse, not better.

Trace-derived content participates in the standard curation lifecycle: low-trust
content decays if never re-accessed, helpfulness scores determine what gets
amplified vs. retired, and aggregation compresses detail once patterns are
extracted. The trace bridge does not generate content that is exempt from
forgetting.

### 5. Trust and provenance are explicit

Every file carries frontmatter declaring its origin, trust level, and
verification status. The harness generates content with:

- `source: agent-generated` — machine-produced from trace data
- `trust: medium` — reliable as a record of what happened, but not user-stated
- `created: YYYY-MM-DD` — date of the session
- No `last_verified` until a human reviews

This means harness-generated session records can be used as context (trust:
medium behavior) but require provenance notice before influencing significant
decisions. They participate in trust decay: session records older than 180 days
without re-access get flagged for review during aggregation.

### 6. Instruction containment

Only `core/memory/skills/` and `core/governance/` may contain procedural
instructions. Plans may contain task-local sequencing for their own scope.
Knowledge files and activity records may *inform* but may not *instruct*.

This applies to everything the harness writes. Session records describe what
happened — they do not prescribe what should happen next. Proposed skills
(Phase 5) are the one place the harness generates procedural content, and they
go through the protected-tier approval process. The trace bridge never writes
instruction-bearing content to `knowledge/` or `activity/`.

### 7. The system improves through use

The feedback loop operates at multiple scales and the harness integration must
feed all of them:

- **Per-retrieval**: ACCESS.jsonl entries with helpfulness scores (from the trace
  bridge's analysis of which file reads led to successful outcomes)
- **Per-aggregation**: Accumulated ACCESS data triggers SUMMARY.md updates,
  identifies enrichment/retirement candidates, and detects co-retrieval clusters
- **Per-session**: Reflection notes capture how memory influenced session quality
  (auto-generated from trace data by the trace bridge)
- **Per-review**: Periodic reviews assess whether the feedback loop itself is
  producing good outcomes (manual, but informed by trace-derived metrics)

The harness doesn't just write data to Engram — it feeds data into a system
designed to learn from it. The trace bridge's output format must match what the
aggregation pipeline expects.

### 8. The user owns the truth

Every change the harness makes to Engram is committed to git with provenance
metadata (commit message, source attribution). The user can inspect, revert, or
delete any harness-generated content at any time via git history. The trace
bridge never produces changes that resist rollback.

Skill proposals (Phase 5) require explicit user approval before promotion to
active skills — this is the protected-tier change model. Session records and
ACCESS entries are automatic-tier (committed without approval) because they are
factual records of what happened, not behavioral changes.

### 9. Engram remains model-agnostic

Engram works with any model that can read text files. The harness calls Engram's
format layer through Python imports — the portable `engram_mcp.agent_memory_mcp.core`
module that exposes frontmatter parsing, git operations, path policies, and
validation without depending on the MCP runtime or any specific model's SDK.

The MCP server continues to run standalone for other clients (Cursor, Claude
Desktop, other agents). The harness is one consumer, not the only consumer.
Nothing about the integration prevents a different agent loop from using the
same Engram repo.

### 10. Graceful degradation

Every integration feature is optional and its absence degrades functionality
without breaking:

- Missing Engram repo → `FileMemory` fallback (flat markdown, no retrieval)
- Missing `sentence-transformers` → keyword search fallback for recall
- Missing trace bridge → sessions work, just don't persist to Engram
- Missing plan infrastructure → sessions are standalone (no multi-session resume)
- Corrupt ACCESS.jsonl → aggregation skips, doesn't crash

Each layer adds value independently. The harness never requires Engram to
function, and Engram never requires the harness to function.

### 11. Maturity-adaptive behavior

Engram models its own developmental stage. A young system (Exploration) uses
loose thresholds and captures aggressively. A mature system (Consolidation) uses
tight thresholds and captures selectively. Active thresholds live in
`core/INIT.md`.

The harness respects the current stage:

- **Exploration**: Trace bridge writes detailed session records, logs ACCESS
  entries generously (lower `min_helpfulness` threshold), and proposes more skill
  candidates.
- **Consolidation**: Trace bridge writes compressed session records, logs only
  high-signal ACCESS entries, and proposes skills only from very strong clusters.

The stage doesn't change the harness's behavior directly — it changes parameters
in the trace bridge's output generation.

### 12. Measure what you mandate

Every piece of bookkeeping the system performs has a cost in tokens and
complexity. Before adding a new recording requirement, ask: does this feed a
feedback loop that measurably improves the system?

The trace bridge generates only data that feeds existing Engram feedback loops:
ACCESS entries (feed aggregation), session records (feed compression hierarchy),
trace spans (feed observability and plan execution), reflections (feed periodic
review). It does not generate governance artifacts, maintenance logs, or
structural metadata that nothing downstream consumes.

### 13. Continuity over novelty

A good long-term agent should feel like it remembers. Cross-session continuity
matters more than single-session cleverness. The harness exists to serve this:
every session contributes to a growing understanding rather than starting from
zero.

This is the thesis of the integrated project. The harness's traces become
Engram's activity history. That history informs future sessions. Future sessions
generate better traces. The loop compounds.

│   ├── loop.py                     # Run loop (model turns → tool dispatch → memory)
│   ├── memory.py                   # MemoryBackend protocol + FileMemory

**Key structural principle:** The harness is the only runtime this repo
ships. Authoritative Engram code — MCP server, CLI, proxy, sidecar — lives
in a separate standalone repo. Here, `harness/_engram_fs/` is the harness's
own copy of the format-layer primitives it needs, and `engram/core/memory/`
is where harness sessions read and write user memory.

- [x] `harness/tools/memory_tools.py`: five tool classes backing the

- [x] Register the suite in `build_session()` when `--memory=engram`.

### Phase 2b — Workspace Tool Surface — **shipped (open items deferred)**

**Goal:** The agent has a mutable operational workspace that sits between
ephemeral scratch and durable memory, with a coherent set of affordances
for threads, working notes, projects, and (eventually) plans.

The design for the expanded surface lives in
`docs/workspace-affordances-draft.md`. It introduces a new `workspace/`
directory peer to `memory/` (no trust frontmatter, no ACCESS tracking,
freely mutable) and a ``work:`` tool family. Tools are registered with
underscore names and documented in the prompt with `work: <op>(...)` or
`work: project.<op>(...)` prefix syntax.

- [x] `harness/workspace.py`: `Workspace` backend (directory layout,
      CURRENT.md parser/writer, project CRUD, SUMMARY.md auto-generation,
      closed-thread archive rotation, project keyword search, plan
      YAML + run-state + postcondition verification)
- [x] `harness/tools/work_tools.py`: 16 tool classes covering status,
      thread, jot, note, read, search, scratch, promote, the seven
      project ops, and the op-dispatched plan tool
- [x] System prompt `## Workspace` section (opt-in via
      `system_prompt_native(with_work_tools=True)`), wired in
      `config.py` when Engram is active
- [x] Thread / project / plan state changes auto-emit `memory_trace`
      events so the trace bridge picks up workspace transitions without
      the agent annotating manually
- [x] `work_search` — project-scoped keyword search over
      `workspace/projects/`, with optional single-project scope
- [x] `work_promote` — graduate a working note into durable memory
      (reuses `EngramMemory.promote_note` for frontmatter + git commit
      with the standard `[chat]` prefix; refuses to overwrite existing
      memory files)
- [x] `memory_context` project parameter — when a project name is
      supplied, its goal and open questions are folded into the
      re-ranking purpose automatically
- [x] `work_project_plan` — op-dispatched create/brief/advance/list
      with flat YAML/JSON under `workspace/projects/<project>/plans/`,
      postcondition verification (`grep:<pattern>::<path>`,
      `test:<command>`, plain text manual), and in-conversation
      approval gates. SUMMARY.md auto-embeds the active plan.
- [x] Retire `plan_tools.py` + decouple the harness from
      `memory/working/`: the legacy four-tool plan surface is gone;
      `_BOOTSTRAP_FILES` no longer includes `memory/working/USER.md`
      or `memory/working/CURRENT.md`; `_active_plan_briefing` and
      `cmd_status._print_active_plans` now scan
      `<workspace>/projects/*/plans/*.run-state.json` against the
      project-root `workspace/` directory (passed in via
      `EngramMemory(..., workspace_dir=...)`). `memory/working` stays
      in `_SEARCH_SCOPES` so `memory_recall` can still surface
      MCP-curated content.
- [x] Promote workspace to a top-level peer of `engram/` and
      `harness/`. Originally lived under `engram/core/workspace/`;
      moved out so the workspace explicitly mediates between the two
      packages rather than belonging to either one. Trace bridge
      ACCESS tracking dropped `workspace/projects` (the workspace is
      ungoverned per the design doc), and EngramMemory accepts the
      workspace location via `workspace_dir` so the integration seam
      stays explicit.
- [ ] MCP-side `memory/working/` retirement. ~45 references across
      `engram/core/tools/agent_memory_mcp/**` (identity_paths,
      plan_approvals, plan_utils, context tools, tool_schemas, etc.)
      still read and write under `memory/working/`. Migrating the
      MCP layer is a separate, larger concern that affects standalone
      Engram users. Out of scope until a dedicated pass.
- [ ] Bootstrap-shrink open question from Phase 2 — whether
      `start_session`'s primer should drop further files in favor of
      agent-initiated `memory_context` / `work_status` calls. Still
      open.

**Exit criteria:** Agent uses `work_status` at session start, tracks
active lines of work as threads, scaffolds projects when scope
warrants, and workspace state changes appear in trace bridge
reflections — all without direct file writes into `workspace/`.

## Open Questions (deferred)

- **Governance integration depth**: How much of Engram's trust/tier enforcement
  should the harness observe vs. leave to the MCP layer? The format layer
  validates path policies already; should the harness also enforce content-size
  guards, trust-boundary guards, and rate limits from the guard pipeline?
  (Deferred per user request.)
- **Multi-user**: The harness is single-user. Engram has nascent multi-user
  support. When team usage matters, the harness will need session identity
  and per-user ACCESS attribution.
- **MCP client mode**: Should the harness call arbitrary MCP servers (not just
  Engram) as agent tools? This would make it a general MCP client, which is a
  different product.
- **Real-time TraceSink**: Post-run trace bridge first (simpler, better
  helpfulness inference). Real-time `EngramTraceSink` later for crash resilience
  (with sidecar-style deduplication against the post-run path).
- **Sidecar relationship**: The sidecar watches Claude Desktop transcripts;
  the trace bridge processes harness traces. They should not overlap. Likely:
  trace bridge is authoritative for harness sessions; sidecar remains for
  platforms that don't have native integration.
- **Browser views**: Engram's dashboard and viewers (`HUMANS/views/`) should
  work with harness-generated session records. This requires that the trace
  bridge output conforms to the viewer parsers' expectations (trace viewer
  format, project viewer plan format, activity timeline format). Not blocked
  but needs validation.

## Migration Path

1. Both projects move into `engram-harness/` preserving internal structure and
   git history (via `git subtree add`)
2. Engram's existing `core/memory/working/projects/agent-harness/` documentation
   becomes the implementation spec for Phase 1
3. `FileMemory` remains the default; `EngramMemory` is opt-in via `--memory=engram`
4. Existing Engram memory repos anywhere on disk are referenced by `--memory-repo`
   path — the merged project doesn't mandate co-location
5. Platform adapters (`CLAUDE.md`, `AGENTS.md`, `.cursorrules`) in the Engram
   subtree continue to work — they point to `core/INIT.md` which remains
   the canonical router

**Goal:** Zero disruption to existing Engram usage across other platforms while
unlocking trace-powered features that only the harness can provide.

## Relationship to Engram's Development Trajectory

The roadmap in `HUMANS/docs/DESIGN.md` § "Part III: Development Trajectory"
describes Engram's independent future directions. Several align directly with
the integrated project:

- **Context budget monitoring** → Phase 6 (retrieval optimization) provides
  this via ACCESS analytics on context loading costs
- **CLI tool** → The harness CLI subsumes several `engram` CLI operations
  (status, search, validate) as natural extensions
- **Import/export tooling** → Trace bridge is effectively an import pipeline
  (JSONL traces → Engram activity records)
