# Review Queue

Pending suggestions for system modifications that the agent could not present to the user in real-time. Items here await user review during a future session.

## Format

### Modification proposals

```
### [YYYY-MM-DD] Brief title
**Type:** proposed | protected
**Description:** What the agent wants to change and why.
**Status:** pending | approved | rejected | superseded
```

If a `proposed` or `protected` item changes system architecture (`core/governance/`, `README.md`, routing/setup prompts, validators, or skills that alter agent operating procedure), include a brief note on its impact to **consistency**, **user-friendliness**, and **context efficiency**.

### Governance proposals

Governance proposals are generated when the agent's periodic review identifies evidence that a governance rule is producing suboptimal outcomes (see `core/governance/security-signals.md` § "Governance feedback"). These are always protected-tier changes — the system generates the insight, but the human approves.

```
### [YYYY-MM-DD] Governance: Brief title
**Type:** governance
**Rule affected:** Which policy/threshold/protocol this proposes to change.
**Evidence:** Quantitative data supporting the change — access patterns, false positive rates, threshold violation counts, re-retrieval rates after archival, etc.
**Current behavior:** What the rule currently does.
**Proposed change:** What the agent recommends instead.
**Architectural impact:** Effect on consistency, user-friendliness, and context efficiency.
**Expected impact:** How this change would improve system behavior.
**Status:** pending | approved | rejected | superseded
```

### Security flags

Security entries are generated by the agent's anomaly detection and instruction-containment systems (see `core/governance/security-signals.md` for anomaly detection and `core/governance/content-boundaries.md` for instruction containment). They represent potential threats, not confirmed attacks — treat them as signals for investigation.

```
### [YYYY-MM-DD] Security: Brief title
**Type:** security
**Trigger:** What anomaly or heuristic raised this flag.
**File:** Path to the flagged file.
**Recommended action:** Review provenance | Reclassify to `core/memory/skills/` | Archive | Investigate access pattern
**Status:** pending | investigated | resolved | false-positive
```

### Lifecycle

Resolved, rejected, superseded, or false-positive items may be moved to an "Archived" section at the bottom of this file after the next periodic review, or deleted. Keeping the last N resolved items is acceptable. The goal is to avoid unbounded growth while preserving recent history for governance evaluation (e.g. false positive rate).

### Triage timing

| Type         | When to surface                                               | User involvement                                                |
| ------------ | ------------------------------------------------------------- | --------------------------------------------------------------- |
| `security`   | **Immediately** — pause current work and present the finding. | Investigate together within the session.                        |
| `protected`  | **Session end** — include in the session-end review.          | Present with reasoning; defer to next session if time is short. |
| `proposed`   | **Session end or next session** — non-urgent.                 | Present with reasoning; user approves or rejects.               |
| `governance` | **Periodic review** — batch with other governance evaluation. | Present with quantitative evidence.                             |

Security flags should never be silently queued for later review. All other types may be deferred if the current session is time-constrained.
