# Projects

`memory/working/projects/` holds one subfolder per active initiative. Each project carries the same skeleton so a cold-starting agent — given only `CLAUDE.md` and this README — can orient in one read.

This file is navigation: "where does X go" and "who writes into Y". Policy lives in `core/governance/`; write this doc should only *cite* governance, never restate it.

## Folder layout at this level

- `SUMMARY.md` — auto-generated projects navigator. One row per project with `Status`, `Mode`, `Open Qs`, `Focus`, `Last activity`. Regenerated by `memory_generate_summary(path="memory/working/projects/")`; on-demand only — can go stale between runs.
- `OUT/` — emitted decisions and artifacts that cross project boundaries. Writes are governed. Contrast with per-project `OUT/`, which is scoped to a single project.
- `ACCESS.jsonl` — append-only access audit for this folder. Consumed by the curation pipeline (`memory_aggregate_access`, `memory_audit_trust`).
- `README.md` — this file.
- `<project>/` — one folder per project.

## Per-project substructure

## Lifecycle

The normal arc for a project:

1. **Create.** `memory_project_create` scaffolds `SUMMARY.md`, `questions.md`, and the empty subfolders above. The creating session is recorded in `origin_session`.
2. **Stage external material.** `memory_stage_external` writes into `IN/` with the snapshot-freshness frontmatter. Reviews happen in `notes/`, not `IN/`.
3. **Plan work.** `memory_plan_create` drafts a plan under `plans/`. Phases are executed with `memory_plan_execute`; verification goes through `memory_plan_verify`.
4. **Iterate on notes.** Agents produce `notes/*.md` as work progresses. `questions.md` grows and gets resolved. `SUMMARY.md`'s `current_focus` and `active_plans` are refreshed as material changes.
5. **Promote durable artifacts.** Mature notes move to `docs/` (citation-grade) via the curation pipeline. Cross-project artifacts go to the projects-level `OUT/`.
6. **Close.** When a project's plans complete, `status` shifts to `completed` (or `abandoned`). The folder is retained; it is not deleted. `cognitive_mode` often shifts from `execution` back to `exploration` as follow-ons surface.

Project folders are never moved laterally — canonical location is `memory/working/projects/<name>/` forever. Renaming is a governed operation and should be rare.

## Conventions

- **Soft over hard.** New frontmatter fields (e.g. `snapshot_taken_at` in P1-B, cold-start fields in `SUMMARY.md` in P1-A) are documented and agent-seeded. Readers degrade gracefully when fields are absent; writers don't reject on missing fields. Exception: plan YAMLs are schema-validated because plan corruption is load-bearing.
- **Agent-maintained briefings.** `SUMMARY.md` at both levels is agent-generated — humans don't hand-author it. Refresh on material change, not on a schedule.
- **`operations.jsonl` is append-only.** Never rewrite, truncate, or compact from an agent. The aggregation pipeline owns compaction.
- **`IN/` is staging, not storage.** Material in `IN/` represents a point-in-time snapshot. When it goes stale, re-snapshot rather than edit in place.
- **`docs/` vs `notes/`.** If you would cite it in a decision, it belongs in `docs/`. If you're still working it out, it belongs in `notes/`. The promotion path is one-way.
- **Two `OUT/` folders, not one.** Project-scoped `OUT/` at `<project>/OUT/`; cross-project `OUT/` at the projects-level `OUT/`. When in doubt: is the consumer inside or outside this project?
- **`memory_context_project_lite` as a safety valve.** If `memory_context_project` trips its time budget twice in a row, or if bootstrap needs guaranteed-fast latency, fall back to `memory_context_project_lite(project=<name>)`. It returns only `SUMMARY.md` + plan IDs but is guaranteed under 500ms.
