# Engram

This repository is a persistent AI memory system. Start with `README.md` for the architectural contract, then continue to `core/INIT.md` for live routing and thresholds. When local agent-memory MCP tools are available, prefer them for memory reads, search, and governed writes; fall back to direct file access only when the MCP surface is unavailable or lacks the needed operation. Do not duplicate the full rule list here — `README.md` and `core/governance/` are the single source of truth.

**Cold-starting on a project.** When the user names a project, call `memory_session_bootstrap` first — active plans come back with `resume_context` pointers ready to dereference. Then read `memory/working/projects/SUMMARY.md` for the navigator of all projects, and `memory/working/projects/<name>/SUMMARY.md` for the per-project briefing (subfolder legend, canonical source, how-to-continue). If you don't know which tool fits, call `memory_route_intent` with what you're trying to do. Per-project `SUMMARY.md` is the agent-maintained briefing artifact — update it when the project's cognitive mode, current focus, or active plans change. See `memory/working/projects/README.md` for the folder lifecycle.

> Note: the former MCP tools `memory_context_project` and `memory_context_project_lite` have been retired from the engram-harness repo. In the harness, their role is filled by `memory_context(project=<name>)`, which folds the project's SUMMARY.md and active plan names into the response. Standalone Engram clients should use `memory_read_file` on the project's `SUMMARY.md` and `plans/` directly, or the equivalent tools in a current standalone Engram build.
